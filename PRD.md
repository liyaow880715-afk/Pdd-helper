## 1. 项目概述

### 1.1 产品背景

拼多多商家在运营多个店铺时面临以下痛点：
- **数据分散**：各店铺订单、商品、售后数据分散在不同后台，需要反复切换登录
- **效率低下**：手动处理订单发货、客服回复、库存监控，人力成本高
- **决策盲区**：缺乏跨店铺的数据聚合分析，难以发现经营问题和机会
- **推广粗放**：多多进宝、搜索推广等投放渠道分散，无法统一追踪 ROI
- **响应滞后**：售后问题、库存预警不能及时发现，影响店铺评分

### 1.2 产品目标

构建一套基于拼多多开放平台的**多店铺统一管理系统**，集成 AI 大模型能力，实现：

1. **多店铺一站式管理** — 单点登录，统一管理 10+ 店铺的订单、商品、售后
2. **AI 智能运营** — 智能客服自动回复、销售数据洞察、推广策略优化建议
3. **推广效果追踪** — 多渠道推广链接统一管理，精准归因分析
4. **自动化预警** — 库存预警、异常订单、退款风险自动通知

### 1.3 目标用户

| 用户类型 | 特征 | 核心需求 |
|---------|------|---------|
| 多店店主 | 运营 3-10 家拼多多店铺 | 聚合看板、批量操作、效率提升 |
| 电商代运营 | 代管多家客户店铺 | 权限隔离、数据报表、客户汇报 |
| 品牌商家 | 有自有品牌，多店分销 | 价格监控、库存统筹、品牌保护 |

### 1.4 核心价值主张

> "一个后台管多店，AI 帮你做运营"

## 3. 功能模块详细设计

### 3.1 店铺管理

#### 3.1.1 多店铺授权接入

**功能描述**：支持通过拼多多开放平台 OAuth2.0 授权，接入多家店铺。

**业务流程**：
```
用户点击"添加店铺"
    → 系统生成拼多多授权链接
    → 跳转拼多多授权页
    → 店铺主登录并确认授权
    → 拼多多回调系统接口（携带 code）
    → 系统用 code 换取 access_token + refresh_token
    → 存储到数据库（shops 表）
    → 店铺显示在管理列表
```

**核心字段**：
| 字段 | 说明 |
|------|------|
| client_id | 开放平台应用 ID |
| client_secret | 开放平台应用密钥 |
| access_token | 店铺授权访问令牌（有效期约 2 小时）|
| refresh_token | 刷新令牌（用于续期 access_token）|
| expires_at | access_token 过期时间 |
| mall_id | 拼多多店铺唯一标识 |

**Token 刷新机制**：
- 定时任务每 30 分钟扫描即将过期的 token
- 调用 `refresh_token` 接口自动续期
- 续期失败（如店铺解除授权）则标记店铺状态为 `disabled`

#### 3.1.2 店铺列表与切换

**功能描述**：展示所有已授权店铺，支持快速切换当前操作店铺。

**页面元素**：
- 店铺卡片：名称、状态（正常/过期/禁用）、授权时间、到期提醒
- 全局店铺切换器：顶部导航栏下拉选择，所有页面共用当前选中店铺
- 操作按钮：刷新 Token、禁用店铺、删除店铺

### 3.2 数据看板 (Dashboard)

#### 3.2.1 核心指标卡

展示当前选中店铺的实时核心数据：

| 指标 | 数据来源 | 更新频率 |
|------|---------|---------|
| 今日销售额 | 本地 orders 表 | 实时（本地数据）|
| 今日订单量 | 本地 orders 表 | 实时 |
| 待发货订单 | 本地 orders 表 | 实时 |
| 库存预警数 | 本地 goods 表 | 实时 |
| 退款/售后数 | 本地 orders 表 | 实时 |
| 推广花费(今日) | 本地 promotion_daily_stats | 每小时 |

#### 3.2.2 销售趋势图

- **近 7 天 / 近 30 天** 销售额与订单量双轴折线图
- 数据来源：本地数据库聚合（避免频繁调用拼多多 API）

#### 3.2.3 热销商品 TOP 10

- 按订单量排序，展示商品名称、销量、销售额
- 支持点击跳转商品详情页

#### 3.2.4 库存预警列表

- 实时展示库存 ≤ 预警阈值的商品
- 红色高亮，支持一键跳转补货

### 3.3 订单管理

#### 3.3.1 订单列表

**筛选条件**：
- 时间范围（今日/近7天/近30天/自定义）
- 订单状态（待发货/已发货/已完成/退款中）
- 关键词搜索（订单号、商品名称、买家信息）

**列表字段**：
| 字段 | 说明 |
|------|------|
| 订单号 | 拼多多订单编号 |
| 商品信息 | 商品名称、规格、数量 |
| 买家信息 | 收货人、手机号（脱敏）、地址 |
| 订单金额 | 实付金额 |
| 状态 | 待发货/已发货/已完成/已退款 |
| 物流信息 | 快递公司、运单号 |
| 下单时间 | 精确到秒 |

#### 3.3.2 订单同步

- **自动同步**：定时任务每 5 分钟拉取新订单
- **手动同步**：页面提供「立即同步」按钮
- **增量同步**：基于 `start_confirm_at` 时间戳，只拉取新增/变更订单

#### 3.3.3 发货操作

- **单订单发货**：填写物流单号，选择快递公司（默认圆通）
- **批量发货**：CSV 导入（订单号 + 运单号），系统批量调用拼多多发货接口
- **发货后自动通知**：飞书推送发货成功消息

### 3.4 商品管理

#### 3.4.1 商品列表

- 展示店铺所有商品：ID、主图、标题、价格、库存、状态（上架/下架）
- 支持按状态筛选、关键词搜索
- 分页展示，每页 20 条

#### 3.4.2 商品编辑

支持修改：
- 商品标题
- 售价 / 拼单价
- 库存数量
- 上下架状态

修改后同步调用拼多多 API 更新，同时更新本地数据库。

#### 3.4.3 批量上下架

- 勾选多个商品，一键批量上架/下架
- 调用 `pdd.goods.status.update` 接口

#### 3.4.4 库存同步

- 每小时自动同步一次全量商品库存
- 库存低于阈值时触发飞书预警通知

### 3.5 客服管理

#### 3.5.1 智能客服（AI 自动回复）

**三级回复策略**：

```
用户消息
    ↓
[第一层] 关键词规则匹配
    → 命中则返回预设回复（0 成本）
    ↓ 未命中
[第二层] AI 缓存匹配
    → 历史相同/相似问题直接返回缓存答案
    ↓ 未命中
[第三层] 大模型生成回复
    → 调用 DeepSeek/Kimi API 生成回复
    → 存入缓存，供下次复用
```

**AI Prompt 设计**：
```
你是拼多多店铺专业客服，回复简洁友好，不超过 150 字。
店铺规则：下单后 48 小时内发货，7 天无理由退换，物流使用圆通快递。

常见问题：
{FAQ}

买家订单信息：
{orderInfo}

客户问题：{message}
```

#### 3.5.2 客服规则配置

- 关键词 → 回复内容 映射表
- 支持优先级设置（多条规则冲突时按优先级匹配）
- 支持通配符/包含匹配

#### 3.5.3 客服消息记录

- 展示所有会话记录：买家消息、回复内容、回复类型（自动/AI/人工）
- 支持按时间、回复类型筛选
- 统计面板：AI 回复率、人工介入次数、常见问题 TOP 10

### 3.6 推广分析

#### 3.6.1 推广计划管理

支持管理两类推广：

| 类型 | 说明 | API |
|------|------|-----|
| 多多进宝 | CPS 推广，按成交付佣金 | `pdd.ddk.*` |
| 推广中心 | 搜索推广 + 场景推广，CPC 竞价 | `pdd.ad.api.*` |

**功能**：
- 同步推广计划列表（计划名称、状态、预算、出价）
- 查看推广单元详细数据
- 推广日报/周报数据聚合

#### 3.6.2 渠道管理

定义投放渠道，用于后续归因分析：

| 渠道类型 | 示例 |
|---------|------|
| 微信生态 | 微信群、朋友圈、公众号 |
| 内容平台 | 小红书、抖音、快手 |
| 站内 | 拼多多直播、多多果园 |
| 其他 | 短信、邮件、线下二维码 |

#### 3.6.3 追踪链接生成

**功能描述**：为每个商品 + 推广计划 + 渠道组合生成唯一追踪链接。

**链接结构**：
```
https://pdd.cc/{short_code}
    ↓ 302 跳转
https://mobile.yangkeduo.com/goods.html?goods_id=xxx
    &utm_source=wechat
    &utm_medium=group_a
    &utm_campaign=plan_123
```

**核心字段**：
| 字段 | 说明 |
|------|------|
| short_code | 6 位短码，用于标识唯一追踪链接 |
| product_id | 商品 ID |
| plan_id | 推广计划 ID |
| channel_id | 渠道 ID |
| target_url | 最终跳转地址 |

#### 3.6.4 效果归因分析

**数据聚合维度**：

| 维度 | 指标 |
|------|------|
| 按渠道 | 点击数、订单数、GMV、ROI |
| 按商品 | 各商品推广效果对比 |
| 按时间 | 7 天 / 30 天趋势 |
| 按计划 | 多多进宝 vs 搜索推广 vs 场景推广 |

**归因逻辑**：
1. **多多进宝订单**：通过 `p_id`（推广位 ID）精确匹配
2. **普通订单**：通过优惠券码 / 时间窗口 + 设备指纹兜底匹配

#### 3.6.5 AI 推广策略建议

基于推广数据，调用大模型生成策略报告：

```
输入：近 7 天推广数据（渠道分布、花费、GMV、ROI）
输出：
1. ROI 分析和问题诊断
2. 各渠道预算分配建议
3. 下一步优化动作（具体可执行）
4. 风险提示
```

### 3.7 AI 数据分析

#### 3.7.1 销售智能分析

**触发方式**：
- 手动触发：页面点击「生成分析报告」
- 自动触发：每天 00:00 生成前一日报告

**输入数据**：
- 昨日销售额、订单量、客单价
- 近 7 天销售趋势
- 热销商品 TOP 10
- 退款率、库存预警

**AI 输出**：
- 数据亮点与异常识别
- 经营建议（定价、选品、促销）
- 需关注的问题预警

#### 3.7.2 日报/周报自动生成

**日报内容**：
```
📊 每日销售日报 (2026-04-30)

💰 销售额：¥12,580
📦 订单量：156 单
🔥 热销商品：商品A-50单、商品B-30单

⚠️ 需关注：
- 商品C 库存仅剩 5 件
- 退款率 2.3%，略高于平均

🤖 AI 建议：
{DeepSeek 生成的经营建议}
```

**推送方式**：飞书 Webhook 自动推送

### 5.3 限流与重试策略

| 策略 | 实现 |
|------|------|
| 单店铺限流 | 每个店铺独立队列，间隔 120ms |
| 限流重试 | 遇到 10038 错误，等待 1s 后重试，最多 3 次 |
| 网络重试 | 连接超时，等待 1.5s 后重试，最多 3 次 |
| 全局限流 | 单 IP 每分钟最多 200 次请求 |

## 6. AI 集成方案

### 6.1 模型选择策略

| 场景 | 推荐模型 | 成本 | 说明 |
|------|---------|------|------|
| 客服对话 | `deepseek-chat` | 低 | 速度快，成本低 |

### 6.2 成本控制

| 措施 | 说明 |
|------|------|
| 关键词优先 | 简单问题匹配规则，不调用 AI |
| AI 缓存 | 相同问题复用历史答案 |
| 每日限额 | 默认 1000 次，可配置 |
| 80% 预警 | 接近上限时飞书通知 |

## 9. 风险与应对

| 风险 | 影响 | 应对措施 |
|------|------|---------|
| 拼多多 API 限流 | 数据同步延迟 | 请求队列 + 指数退避重试 |
| API 权限审核不通过 | 功能无法使用 | 分批次申请，先基础后高级 |
| AI 调用成本超支 | 运营成本增加 | 关键词优先 + 缓存 + 每日限额 |
| Token 过期未刷新 | 店铺断开连接 | 定时任务 + 过期预警 |
| 数据安全泄露 | 店铺信息泄露 | Token 加密存储、日志脱敏、HTTPS |
